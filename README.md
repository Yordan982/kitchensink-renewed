# Kitchensink Modernization (Spring Boot + MongoDB)

## Overview

This project is a modernization of the legacy *kitchensink* application from the JBoss EAP Quickstarts repository.

The goal was to migrate a traditional Java EE application to a modern, cloud-ready architecture using:

* **Spring Boot (Java 21)**
* **MongoDB**
* RESTful APIs
* Improved validation, error handling, and testing

The migration preserves the original functionality while improving maintainability, scalability, and developer experience.

---

## Modernization Approach

Instead of a direct rewrite, the migration followed a structured approach similar to what would be used in a larger enterprise system:

### 1. Understand the Legacy Application

* Analyzed the original JBoss application structure (CDI, JPA, JSF)
* Identified the core domain: Member registration and listing

### 2. Define Target Architecture

* Replaced Java EE stack with **Spring Boot**
* Switched from relational DB to **MongoDB** (document model)
* Introduced **REST API layer**
* Introduced a new simple UI to focus on the backend modernization

### 3. Incremental Migration Strategy

* Recreated the domain model first
* Implemented API endpoints
* Added validation and error handling
* Introduced pagination and search
* Added tests to ensure code accuracy

This approach can be extended to larger systems by migrating components incrementally, allowing coexistence between legacy and modern services during the transition.

---

## Key Improvements

### 1. Simpler Architecture

* Removed heavy Java EE dependencies
* Replaced with lightweight Spring Boot stack

### 2. MongoDB Integration

* Switched from relational schema to document-based model
* Used:

    * `@Document`
    * `@Indexed(unique = true)` for email uniqueness

### 3. Validation Improvements

* Bean Validation (Jakarta Validation)
* Custom validator for email uniqueness
* Input normalization (trim, lowercase)

### 4. Error Handling

* Centralized exception handling via `@ControllerAdvice`
* Consistent API error responses:

```json
{
  "timestamp": "...",
  "status": 400,
  "message": "Validation failed",
  "errors": {
    "email": "email must be valid"
  }
}
```

### 5. Pagination & Search

* Pagination with sensible defaults and limits
* Search across multiple fields using MongoDB queries
* Protected against regex injection using `Pattern.quote`

### 6. Testing

Unit tests were created for:
* Custom validator
* Service layer
* Controller layer

The tests focus on behavior and edge cases rather than simple data structures.

## How to Run the Application

### Prerequisites

Make sure the following are installed:

- Java 21
- Maven 3.8+

### 1. Clone the repository

```bash
git clone https://github.com/Yordan982/kitchensink-renewed.git
cd kitchensink-renewed
```

### 2. Configure MongoDB

Update `application.properties` with a MongoDB database:

```properties
spring.data.mongodb.uri=<mongodb-uri>
```

### 3. Build the project

```bash
mvn clean install
```

### 4. Run the application

```bash
mvn spring-boot:run
```

### 5. Access the application

* UI: http://localhost:8080
* API: http://localhost:8080/api/members

## API Endpoints

### Create Member

```
POST /api/members
```

### Get All Members (optional pagination & search)

```
GET /api/members?page=1&size=5&search=Yordan
```

### Get Member by MongoDB document ID

```
GET /api/members/{id}
```

## Design Decisions

### Why MongoDB?

* Its flexible schema fits the evolving domain
* Simplifies data model vs legacy relational mapping
* Aligns with modernization goals of AMP

### Why Keep Both Validator + DB Constraint?

* Validator means fast user feedback
* DB constraint guarantees consistency under concurrency
